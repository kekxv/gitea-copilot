# GiteaCopilot - Gitea 专属智能化 Agent 服务 (PRD & Tech Spec)

## 1. 项目概述
本项目（GiteaCopilot）是一个轻量级的独立后端服务，通过 Webhook 与 Gitea 深度集成。当用户在 Gitea 的 Issue 或 Pull Request 中 `@机器人账号` 时，系统会自动捕获消息，利用 LLM 执行代码分析、自动打标签、文档问答等任务。

系统支持多租户/多 Gitea 实例设计，具备自动化的 Webhook 管理、高精度的代码审查挂载能力以及严苛的安全脱敏机制。

## 2. 核心系统角色
*   **管理员 (Admin)**：负责系统全局配置（LLM API、系统限制）、查看服务状态，采用 Password + TOTP 双因素认证。
*   **普通用户 (User)**：通过 Gitea OAuth2 登录，支持多 Gitea 实例管理，负责仓库授权及 Skills 配置。

## 3. 核心功能与技术实现规格

### 3.1 账号与自动化管理
*   **Token 生命周期管理**：集成 `APScheduler` 实现 `access_token` 的自动检查与 `refresh_token` 无感刷新。
*   **Webhook 自动化**：一键授权模式。系统通过 Gitea API 自动配置 Webhook，并利用 `Custom Header` 写入 Base64 编码的用户上下文，实现多实例请求的精准识别。

### 3.2 智能化代码审查 (Code Review) - [深度优化项]
*   **精准挂载技术**：
    *   **字段对齐**：使用 Gitea Review API 专属的 `new_position` 和 `old_position` 字段，而非普通评论的 `line`。
    *   **版本锁定**：强制在请求中携带 PR 的 `head_sha` (Commit ID)，确保评论精准锚定在正确的提交版本上。
    *   **白名单过滤**：系统实时构建 Diff 行号白名单 (`valid_lines`)，过滤 AI 幻觉产生的无效行号，防止 Gitea 渲染崩溃。
*   **语义一致性审计**：
    *   **注释-代码对齐**：检查注释是否准确描述了代码逻辑，识别过时或误导性描述。
    *   **命名-功能对齐**：检查函数名、变量名是否与其实现的逻辑功能相符。
*   **分块处理 (Batching)**：支持大 PR 自动拆分。按文件或 Hunk 组划分区块（约 800-1000 行），循环调用 LLM 并汇总最终报告，突破上下文长度限制。

### 3.3 安全审计与隐私保护
*   **专项凭证扫描**：AI 强制扫描变更代码中的硬编码 Token、Key、密码、公私钥等敏感信息。
*   **极致脱敏逻辑 (`scrub` 函数)**：
    *   **第一层 (AI 强指令)**：严禁 AI 在评论正文中回显具体的敏感值。
    *   **第二层 (系统级正则)**：在提交至 Gitea 前，对 Summary 和 Comments 进行全词匹配（使用 `\b`）脱敏，涵盖 20+ 种敏感词特征，确保格式不乱码、加粗不失效。

### 3.4 交互体验逻辑
*   **LGTM 自动化**：若全部分块无行评论且 AI 判定无风险，自动回复精简的 `LGTM`。
*   **失败引用机制**：若 AI 调用或系统处理失败，以 Markdown 引用格式回复原指令并告知详细原因。
*   **防死循环策略**：采用“提及屏蔽”技术（在 `@帐号` 后插入空格），在支持“本人 @ 自己”单帐号模式的同时，物理阻断 Webhook 触发。

## 4. 健壮性设计
*   **原子化幂等性**：利用数据库唯一索引 (`reference_id`) 配合 `try_record_event` 事务逻辑，消除高并发下的重复处理竞争。
*   **AI 幻觉引导**：针对 AI 返回的非规范 JSON 格式，增加明确的错误指令反馈，引导 AI 自动纠错并重新生成。

## 5. 技术栈与架构约定
*   **后端**：Python 3.10+ & FastAPI
*   **环境/包管理**：`uv` (推荐) 或 `pip`
*   **存储**：SQLite + SQLAlchemy (原子事务控制)
*   **任务分发**：FastAPI 原生 `BackgroundTasks` (轻量化异步)
*   **报告格式**：Markdown 结构化输出（表格展示状态 + 列表展示统计）。

## 6. 安全与隐私强制准则
1.  **全量脱敏**：任何涉及 Key、Token 的内容严禁出现在 Gitea 界面。
2.  **身份隔离**：严格校验 Webhook 签名 (HMAC-SHA256) 与自定义 Header 上下文。
3.  **不回显敏感值**：即使在报错信息中，也必须对原消息进行脱敏。
